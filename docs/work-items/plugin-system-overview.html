Design Spec v1

TokenBar Plugin System

让社区为任意 AI 编码工具编写数据源适配器，通过 GitHub Registry 一键分发，TokenBar Gallery 一键安装。

Plugin Types
两层插件设计
Level 1

Declarative 声明式

纯 JSON manifest，零代码

  • JSONL dot-notation 字段映射
  • SQLite 表名 + 列映射 + WHERE
  • 复用现有 GRDB / JSONL 引擎
  • 安全：只有配置，无可执行代码
覆盖 ~80% AI 工具（GitHub Copilot、Kiro、Qwen Code、Kimi ...）
Level 2

Executable 可执行

外部脚本/二进制，输出 NDJSON

  • Python / Node / Bash / 任意语言
  • TokenBar spawn 子进程 + 读 stdout
  • 支持 --since 参数实现增量
  • 安装时二次确认 + timeout 保护
覆盖 ~20% 需要转换逻辑的工具（Cursor、远程 API ...）

Executable Protocol
可执行插件通信协议
TokenBar ↔ Plugin Script 约定
调用
{command} {script} {args} --since 2026-05-27T00:00:00Z
stdout
NDJSON — 每行一个 UsageEvent，必填：id, timestamp, input_tokens, output_tokens
stderr
警告/错误信息 → 采集为 ParseWarning，不影响正常事件
退出码
0 = 成功  |  非 0 = 失败（记录 warning，下次重试）
环境变量
TOKENBAR_PLUGIN_STATE_DIR → ~/Library/.../plugins/{id}/state/ 脚本可自行管理状态
超时
默认 30s，manifest 可配，上限 120s → Process.terminate()
Token Normalization
inputIncludesCached 归一

借鉴 ali-trace：不同 provider 的 input_tokens 口径不一致。Anthropic/OpenAI 的 input 已含 cache_read， 其他 provider 的 input 是净值。manifest 中 input_includes_cached 标志声明口径，归一层统一处理。

原始数据 (inputIncludesCached: true)

rawInput5000
cacheRead3000
output1200
cacheCreation800
reasoning0
→

归一后写入 DB

input2000 ← 5000 - min(3000, 5000)
cacheRead3000
output1200
cacheCreation800
reasoning0
Integration
数据流：从 Manifest 到 usage_events
manifest.json
Registry 下载 → ~/Library/.../plugins/{id}/manifest.json
PluginLoader.parse(manifest)

CustomSourceRegistry.upsert(record)
写入 custom_sources 表 · pluginId 关联 · sourcePathKey 去重
CheckpointEngine 调度
jsonl → parseMappedJSONL() · sqlite → PluginSqliteReader · exec → PluginExecutableRunner
TokenNormalizer.normalize()
clamp 负值 → 0 · inputIncludesCached → input -= cacheRead
usage_events 写入
id 前缀 "plugin:{id}:" · watermark 更新 · 即刻出现在 UI/CLI

Roadmap

Security
安全设计
声明式 — 安全

零执行风险

  • 纯 JSON 配置，无可执行代码
  • GRDB 只读模式，无写入能力
  • 字段映射不执行表达式
  • SQL 参数化 + 表/列名白名单
可执行 — 受控

多层防护

  • Registry PR 审核脚本内容
  • 安装时二次确认弹窗
  • timeout 硬上限 120s
  • 全本地数据，无上传通道
